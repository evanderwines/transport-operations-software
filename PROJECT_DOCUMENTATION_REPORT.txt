======================================================================
2. EXECUTIVE SUMMARY
======================================================================
This repository is a Laravel 12 + Inertia + React 19 transport operations
application with three main user roles:
- ADMINISTRATOR
- DRIVER
- CUSTOMER

The core business domains are:
- user management
- reservation creation and tracking
- dispatch assignment and monitoring
- vehicle and driver fleet management
- driver task execution
- customer reservation self-service
- system log reporting
- real-time vehicle location updates

The app behaves like a role-aware operations dashboard rather than a purely
traditional CRUD system. Most user-facing screens are rendered through Inertia,
with Laravel controllers providing server-driven data and React pages/components
handling client interactivity.

======================================================================
4. HIGH-LEVEL ARCHITECTURE
======================================================================
Application shape:
- Laravel handles HTTP routing, authentication, validation, data access, and Inertia responses.
- React handles page rendering, forms, dashboard visuals, filtering UX, maps, and live updates.
- Sanctum protects most application routes with auth:sanctum.
- An httpOnly auth_token cookie is attached as a Bearer token by custom middleware.
- Echo/Reverb is used for broadcasting reservation and vehicle location events.

Request flow:
1. Browser request enters Laravel.
2. AttachTokenFromCookie middleware copies auth_token into Authorization when needed.
3. auth:sanctum resolves the user.
4. Controller queries Eloquent models and returns Inertia payloads.
5. React page renders the role-specific screen.
6. Optional Echo subscriptions update the UI in real time.

Role split:
- Administrators get dashboards plus users, reservations, fleet, active dispatches, and logs.
- Drivers get tasks, task details, live location broadcasting, and a dashboard.
- Customers get their own reservations, active reservations, and a dashboard.

======================================================================
5. AUTHENTICATION AND AUTHORIZATION MODEL
======================================================================
Auth implementation is customized beyond the default starter kit.

Important behavior:
- resources/js/pages/auth/login.tsx posts to /api/login instead of the default web login form route.
- routes/api.php issues a Sanctum personal access token and stores it in an httpOnly auth_token cookie.
- app/Http/Middleware/AttachTokenFromCookie.php injects that cookie into the Authorization header.
- bootstrap/app.php prepends AttachTokenFromCookie globally so it runs before auth middleware.
- app/Http/Middleware/HandleInertiaRequests.php shares auth user data and optional auth_debug props.
- routes/web.php contains debug endpoints when APP_DEBUG is enabled.

Role-based behavior:
- Dashboard routing is role-aware in app/Http/Controllers/DashboardController.php.
- Sidebar navigation is filtered by role in resources/js/components/ui/sidebar.tsx.
- Reservation index redirects customers to /my-reservations.

Operational note:
- The project currently mixes starter-kit web auth conventions with API token auth conventions.
  The visible login UI uses /api/login, while many default auth tests still target login.store.

7. DOMAIN MODEL AND DATA STRUCTURE
======================================================================
Core models:
- User
  - Standard auth entity.
  - Stores role and role_id.
  - CUSTOMER users relate to reservations through customer_id.
- Reservation
  - Primary business record.
  - String primary key reservation_id.
  - Stores status, addresses, lat/lng strings, date/time, service type, cargo, instructions.
- Dispatch
  - Links reservation to vehicle and schedule.
  - schedule and assigned_at drive operational planning.
- Driver
  - Separate entity keyed by driver_id.
  - Vehicle assignment is based on driver_id.
- Vehicle
  - Keyed by vehicle_id.
  - Stores assigned driver, plate, model, capacity, status.
- SystemLog
  - Stored in logs table.
  - Tracks datelog, timelog, action, module, performed_to, description.

Relationship summary as implemented:
- User hasMany Reservation via customer_id.
- Reservation belongsTo User as customer.
- Reservation hasOne Dispatch.
- Driver hasOne Vehicle.
- Vehicle belongsTo Driver.
- Dispatch exposes vehicle via hasOne(Vehicle::class, 'vehicle_id', 'vehicle_id').
- Vehicle exposes dispatch via belongsTo(Dispatch::class, 'vehicle_id', 'vehicle_id').

Important modeling note:
- The dispatch/vehicle relationship methods are unconventional for a historical dispatch model.
  In practice, dispatches point at vehicles via vehicle_id, but the relationship types suggest a tighter
  one-to-one view than the business domain may need.

Role linkage note:
- Driver dashboards/tasks infer the driver entity through users.role_id == drivers.driver_id.

Common reservation status values seen in code and seed data:
- PENDING
- ASSIGNED
- GOING TO PICKUP
- WAITING
- GOING TO DROPOFF
- EN ROUTE
- COMPLETE

Reservation creation flow:
1. Step 1 selects vehicle/date.
2. Step 2 captures pickup location.
3. Step 3 captures dropoff location.
4. Step 4 captures service details.

18. OBSERVATIONS, RISKS, AND TECHNICAL DEBT
======================================================================
1. Authentication approach is mixed.
   - The repo keeps both starter-kit auth routes and a custom /api/login token-cookie flow.
   - This is workable, but it increases cognitive load and testing complexity.

2. Hardcoded external credentials/endpoints exist in source.
   - resources/js/bootstrap.ts hardcodes websocket host/key.
   - resources/js/lib/utils.ts hardcodes GraphHopper API key.
   - config/reverb.php contains explicit app key/secret/host values.
   - These should move to environment configuration.

3. Missing event class reference.
   - app/Http/Controllers/Driver/LocationController.php references DriverLocationUpdated,
     but no corresponding event file was found in app/Events.

4. Empty migrations directory.
   - database/migrations is empty, so schema evolution is not documented in migration source.

5. Some type and model inconsistencies exist.
   - resources/js/types/index.d.ts defines Vehicle twice.
   - Dispatch seed data includes status, but Dispatch model fillable fields omit it.
   - Relationship semantics between Dispatch and Vehicle are unconventional.

6. Admin user deletion is not fully logged.
   - UserController::destroy deletes users but does not write a system log entry.
   - ProfileController::destroy does log self-deletion.

7. Potential null-safety issue in driver tasks.
   - TaskController@index assumes every driver has an assigned vehicle before dereferencing first()->vehicle_id.

8. UI/state cleanup is uneven.
   - user-menu-content.tsx still clears localStorage auth_token even though auth is primarily cookie-based.

9. Some screens still show starter-kit residue, typos, or incomplete UX wiring.
   - Example: 'Active disptaches' typo in admin active-dispatches page.
   - System logs page computes filter state and actions, but visible filter controls are not currently rendered.
   - Several files contain unused imports and residual placeholder patterns from the original starter kit.
